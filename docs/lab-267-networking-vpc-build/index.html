Build Your VPC and Launch a Web Server

This lab outlines the process of building an entire Virtual Private Cloud (VPC) architecture from scratch to satisfy a customer design request. It combines routing, security perimeters, and automated EC2 bootstrapping.

Lab Objective Create a VPC, configure public/private subnets within a single Availability Zone, provision a Security Group, and automatically deploy an Apache Web Server upon boot.
Tools & Concepts
Amazon VPC Subnets & Routing Security Groups EC2 User Data
Key Note on AZs Although VPC architecture often uses multiple Availability Zones for High Availability, all resources (Public Subnets 1 & 2, Private Subnets 1 & 2) in this lab were deliberately deployed to a single Availability Zone (us-west-2a).

Lab Summary

The network was built starting from the VPC base CIDR block, moving down to subnet allocation, assigning route tables, securing with a Security Group, and finally launching compute resources into the custom network.

Build Process

01

Provisioning the Base VPC & Subnets

The foundation was established using the AWS "VPC and more" wizard, which automatically handled the creation of the primary routing resources.

  • VPC: Lab VPC (IPv4 CIDR: 10.0.0.0/16)
  • Availability Zone: 1 AZ
  • Primary Subnets: 1 Public Subnet, 1 Private Subnet in us-west-2a.
  • Created additional subnets manually (Public Subnet 2, Private Subnet 2), all located in the exact same Availability Zone (us-west-2a) rather than splitting across multiple AZs.
02

Configuring the Security Group

Before launching instances, a virtual firewall was defined to protect the web tier.

  • Created the Web Security Group specifically attached to the Lab VPC.
  • Added an Inbound Rule for HTTP traffic from Anywhere IPv4 (0.0.0.0/0).
03

Bootstrapping the Web Server via EC2 User Data

An EC2 instance was launched directly into the newly created Public Subnet 2. To automate the software setup, a startup bash script was provided in the Advanced Details section.

#!/bin/bash
# Install Apache Web Server and dependencies
yum install -y httpd mysql php

# Download Lab files
wget https://aws-tc-[...]/lab-app.zip
unzip lab-app.zip -d /var/www/html/

# Turn on web server daemon
chkconfig httpd on
service httpd start
04

Final Verification

After the automated installation completed, passing the 2/2 status checks, the server was tested via its Public IPv4 DNS.

Key Learnings

Technical Concepts

✓
Route Table Associations: Creating a subnet is not enough. You must manually associate new subnets with the correct Route Table (Public vs. Private) for them to inherit internet accessibility.
✓
Automation via User Data: Sending scripts via User Data on EC2 launch eliminates the need for manual SSH sessions when setting up web servers.

Architecture Insight

While this lab successfully built a functioning web architecture, it prioritized simplicity by housing all subnets and instances into a single Availability Zone (us-west-2a).

In a true production environment for a Fortune 100 client, these subnets should be distributed across multiple Availability Zones to ensure High Availability and fault tolerance against localized data center outages.
